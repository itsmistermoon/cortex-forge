Qué es la suite de skills de OpenKnowledge

OpenKnowledge es una plataforma de colaboración markdown-CRDT: convierte un directorio de .md/.mdx en una base de conocimiento multi-escritor en tiempo real, expuesta vía un servidor MCP con 20 herramientas (exec, search, write, edit, lint, workflow…). Sus skills no son la funcionalidad — son el contrato de uso que gobierna cómo un agente debe usar esas herramientas.

Arquitectura en tres niveles + dos utilitarias

  • Nivel descubrimiento — open-knowledge-discovery: qué es OK, cómo instalarlo (ok init), compartirlo, abrir archivos sueltos. Su description prohíbe explícitamente usarla para leer/escribir ("Do NOT load to perform reads/writes") y delega al nivel siguiente.
  • Nivel plataforma — open-knowledge (la skill "project"): el contrato de runtime autoritativo. Instalada por ok init como skill project-local. Es un router: 133 líneas de MUSTs + tabla de 20 herramientas + 14 references de profundidad cargadas bajo demanda.
  • Nivel packs — 8 skills instaladas por ok seed --pack X, una por "forma de vault" (wiki de código, CRM personal, base de conocimiento, notas planas, ciclo de vida de software, worldbuilding, pipeline de escritura, formato OKF). Todas repiten el mismo pie: "Complements the platform skill; does not replace it" — se apilan sobre el contrato, nunca lo sustituyen.
  • Utilitarias — bug-report (diagnóstico vía CLI) y write-skill (meta-skill: cómo escribir skills, con evaluación RED/GREEN y pressure-testing).

Las cinco decisiones de diseño que definen la suite

  • Model-invoked por diseño. Ninguna skill es user-only: todas confían en que su description (377–691 caracteres, optimizada como gatillo) dispare la carga automática. Tienen un reference completo (description-optimization.md) dedicado a esta ingeniería.
  • La lógica vive en el servidor, no en el prompt. Lint, auditoría de links, guías de workflow, plantillas: todo son herramientas MCP. Las skills solo enseñan cuándo y cómo llamarlas.
  • Regla STOP como columna vertebral: prohibido usar herramientas nativas (Read/Grep/Edit/rm) sobre markdown del proyecto — todo pasa por MCP para no romper el CRDT ni perder atribución. La regla anticipa y cierra una por una las racionalizaciones del modelo.
  • Divulgación progresiva estricta: el núcleo lleva los MUSTs, los references llevan la profundidad ("Core carries the MUSTs… this file carries the detail"), un solo nivel.
  • Grounding en circuito cerrado: toda afirmación cita una fuente que vive dentro de la KB (las URLs externas se ingieren primero); una URL suelta es un TODO, no una cita.
TL;DR comparativo: OpenKnowledge es un contrato para operar un servidor (skills = política sobre herramientas MCP); Antu es una metodología autocontenida (skills = pipelines con scripts locales); Kuyen es la misma metodología destilada a puro juicio (skills = prosa sin dependencias). Los tres convergieron de forma independiente en varias convenciones — la más notable, el log.md append-only con entradas datadas.

Arquitectura y relaciones

OpenKnowledge — hub & spoke sobre el servidor MCP

Skill núcleo Pack Infraestructura (no skill) — línea sólida: delegación explícita · punteada: "complementa"

Claves de la topología OK: (1) jerarquía con delegación explícita por nombre — discovery manda al agente a la skill project-local, la plataforma manda con un "STOP and invoke" a write-skill; (2) anti-solapamiento diseñado: las descriptions incluyen anti-gatillos ("Do NOT load to…") para que dos skills no compitan por la misma situación; (3) los packs nunca se llaman entre sí — cada vault instala uno solo, y todos se apoyan en el mismo contrato de plataforma; (4) la dependencia dura no es entre skills sino de todas las skills hacia el servidor MCP (ninguna es autosustentable sin él, salvo discovery y bug-report que operan por CLI).

Antu/Kuyen — pares en pipeline, coordinados por el usuario

Antu Kuyen — sólida: una skill propone invocar a la otra · punteada: comparten artefacto/estado

En Antu no hay hub: son pares que comparten artefactos (el vault, .hot/, el config global) y se recomiendan mutuamente (recall → imprint/ingest, prune → ingest, handoff → triage), pero ninguna invoca a otra automáticamente — siempre proponen y el usuario dispara el slash command. La interoperabilidad más sofisticada es entre suites: los marcadores suite: antu / suite: kuyen en HANDOFF.md hacen que cada handoff detecte y archive con cortesía el estado del otro, y antu-triage sabe recuperar pendientes desde bloques archivados de Kuyen.

Estilo de redacción

Rigidez de la prosa (estimación cualitativa por skill, 0 = prosa libre, 100 = contrato estricto)

OpenKnowledge: lenguaje de contrato con gradiente deliberado

  • Registro RFC: "Every rule below is a MUST unless marked otherwise". MUST/SHOULD/NEVER/STOP en mayúsculas; la skill de plataforma concentra la rigidez (8 MUST, 6 NEVER, 6 STOP en 133 líneas).
  • Comandos exactos, no instrucciones generales: formas de llamada literales con shapes JSON — write({ document: { path, template } }), edit({ skill: { name, file, find, replace } }) — más comandos bash textuales y formatos de entrada de log con plantilla exacta.
  • Ingeniería anti-resquicio: la regla STOP enumera y cierra racionalizaciones una a una ("Not seeing exec is NOT the escape hatch", "Absence from the visible list means 'I have not discovered it yet'"), y define la única válvula de escape con precondiciones verificables y una frase obligatoria ("OpenKnowledge MCP unavailable:").
  • Tablas Don't/Do de anti-patrones, con versión resumida en el núcleo y completa en reference.
  • El gradiente es una decisión consciente: write-skill lo formaliza como "degrees of freedom" (alto = prosa libre para juicio, bajo = script fijo para tareas frágiles). Y se nota: la plataforma es contrato duro; plain-notes termina en "write freely".
  • Prosa densísima en la plataforma: paréntesis anidados, incisos, negritas estructurales — optimizada para densidad de información por token, no para lectura humana cómoda.
«Route every in-scope markdown read and write through OK's MCP tools — never your host's native file tools. […] The ban covers every common rationalization:»

Antu: procedimiento numerado con criterios de término

  • Esqueleto fijo por skill: flavor line localizada → Available scripts → Steps numerados → Output format → Rules. Segunda persona imperativa.
  • Criterios verificables: "Done when:" (parcial: handoff, ingest, triage), gates de confirmación explícitos, caps duros numéricos (máx. 20 candidatos, cap de 5 pendientes, 15 días de rotación), severidades HIGH/MEDIUM/LOW.
  • Fail-loud codificado: el fail-open de la sanitización exige avisar ("do not proceed silently"); handoff obliga a declarar "Safe to end" incluso cuando todo está bien ("silence should never be read as an implicit all good").
  • Anti-resquicio al estilo OK en puntos críticos: "Never reconstruct a credential… not even on explicit user insistence"; recall descalifica el conocimiento paramétrico "even if you believe you already know the answer".
  • Comandos exactos cuando hay script (bash scripts/antu-prune.sh {vault}), instrucciones parametrizadas cuando hay juicio.

Kuyen: la misma disciplina, destilada

  • Pasos numerados donde cada paso cierra con "Done when…" — la convención que en Antu es parcial, en Kuyen es sistemática (14 "Done when" en 4 skills vs 8 en 7 skills de Antu).
  • Cero comandos, cero scripts, cero references: pura instrucción de juicio ("Use judgment on what counts as 'the same thing'; don't chase superficial overlap").
  • Prosa más narrativa y corta (~400 palabras/skill vs ~1.100 de Antu y ~1.500 del núcleo OK).

Las descriptions delatan el modelo de invocación

Las descriptions de OK son gatillos largos (377–691 caracteres) escritos "en las palabras del usuario", con anti-gatillos incluidos — porque de ellas depende que la skill se cargue sola. Las de Antu/Kuyen son más cortas y descriptivas: con disable-model-invocation: true en todas, la description es documentación para el humano que teclea el slash command, no una superficie de disparo. Antu invierte lo que OK gasta en descriptions en otra cosa: argument-hint, flavor lines localizadas y resolución multi-vault por argumento.

Duplicados y especialización

En OpenKnowledge: cero duplicados funcionales, mucha repetición deliberada de plantilla

  • No hay dos skills para la misma tarea. El par más cercano es knowledge-base vs okf (ambos "KB genérica"), pero se especializan en ejes distintos: kb = disciplina de evidencia en 3 capas con los workflow tools; okf = conformidad con el formato OKF de Google (portabilidad).
  • El anti-solapamiento está institucionalizado: write-skill lo impone como gate de la Etapa 1 ("If an existing skill covers most of it, STOP and recommend reuse — a near-duplicate with overlapping triggers mis-fires and dilutes both"). O sea: tienen una regla escrita contra los duplicados, aplicada por su propia meta-skill.
  • discovery vs project se solaparían naturalmente (ambas "sobre OpenKnowledge") — lo resuelven con anti-gatillos cruzados en las descriptions y párrafos "Working inside a project — use the project-local skill, not this one".
  • Repetición de plantilla, no de función: 8 packs comparten el mismo esqueleto y frases idénticas ("Complements the platform skill; does not replace it", "Templates carry only structure; meaning lives here, not in the document body"). Es boilerplate asumido como costo de distribución — el mismo trade-off que Antu con VAULT-RESOLUTION.md copiado en 6 skills.
  • La especialización "uno preserva, otro sintetiza" existe pero no como skills: son los kinds del workflow tool — ingest (verbatim) → research (provisional) → consolidate (canónico con cadena supersedes:) — con transiciones que el usuario debe conducir ("Don't chain silently").
  • Matiz semántico de log.md entre packs: knowledge-base pide una entrada por turno, codebase-wiki una por corrida, okf una por cambio — misma convención, cadencias distintas por dominio. Variación intencional, pero es el tipo de deriva que en tu ADR 0001 unificaste a propósito.

En Antu/Kuyen: especialización por origen del contenido y por objeto de mantención

  • antu-ingest vs antu-imprint — la pareja "uno escribe desde fuentes externas, otro sintetiza la sesión". La frontera es el origen: URL/archivo con .raw/ vs conversación con test de síntesis circular. Kuyen demuestra que la fusión es viable (kuyen-ingest acepta ambos), al costo de perder la jerarquía de fuentes y la sanitización.
  • antu-prune vs antu-triage — misma tarea (mantención) sobre objetos distintos: contenido del vault vs estado de sesión en .hot/. Sin solapamiento real; se tocan solo en el nudge de handoff.
  • Antu vs Kuyen completo — el "duplicado" es la suite entera, y es explícitamente deliberado: full vs lite por vault, documentado en el README de Kuyen con tabla de decisión y convenciones compartidas rastreadas en docs/family-conventions.md. OK no tiene un equivalente de esto (no hay "OK lite").

Lectura de fondo. No compiten en el mismo plano: OK vende un motor (CRDT + MCP) y usa las skills para que el agente no lo rompa; Antu/Kuyen venden un método y usan las skills como único ejecutor. Por eso OK puede permitirse skills model-invoked con descriptions agresivas (el costo de un misfire es bajo: el servidor valida todo), mientras Antu elige user-only (sin servidor que valide, el usuario es el gate). La consecuencia inversa: OK necesita ~20.000 palabras de contrato para contener al modelo; Kuyen logra la misma disciplina de grounding en 1.600 palabras porque renuncia a la infraestructura que hay que proteger. Antu queda exactamente al medio — y su par full/lite (Antu/Kuyen con marcadores de suite cruzados) es un patrón de diseño que OK no tiene.

Hallazgos extra — lo que se te podría estar escapando

Convergencias independientes (validan tus decisiones)

  • log.md append-only datado — los tres sistemas llegaron solos a la misma convención (OK lo llama "log discipline (MUST)"; okf lo hereda del estándar de Google). Tu decisión de ADR 0001 está corroborada por convergencia externa.
  • "Templates llevan estructura; el significado vive en la skill" — OK lo repite en 6 packs; es exactamente tu separación templates/ vs SKILL.md.
  • "Ofrecer persistir, rara vez" — corpus-qa de OK y tu antu-recall/kuyen-query tienen la misma regla con casi la misma redacción (solo cuando combina 2+ fuentes o llena un vacío real; nunca como footer permanente).
  • Boilerplate duplicado como costo de distribución — OK copia frases idénticas en 8 packs; tú copias VAULT-RESOLUTION.md en 6 skills. Nadie resolvió la des-duplicación de guía compartida entre skills instalables.

Ideas robables de OK

  • Pressure-testing formalizado (write-skill, etapas 4/6): baseline RED sin la skill capturando las racionalizaciones textuales del modelo, luego GREEN con 2–3 presiones combinadas (tiempo, autoridad, costo hundido). Tus reglas anti-resquicio ("not even on explicit user insistence") nacieron de intuición; OK tiene el método para fabricarlas sistemáticamente. Candidata directa a incorporarse a tu metodología de desarrollo de skills.
  • Una meta-skill de autoría — no tienes equivalente de write-skill: algo que gobierne cómo se escriben/mejoran las skills antu/kuyen (gate anti-duplicados incluido).
  • "Degrees of freedom" explícitos — clasificar cada skill (o cada paso) por cuánta libertad admite: prosa libre / pasos parametrizados / script fijo. Tú ya lo haces de facto (prune capa 1 = script fijo, capa 2 = juicio); nombrarlo permitiría auditarlo.
  • Persistencia incremental como MUST — "la KB es tu checkpoint": en tareas largas, escribir cada unidad al terminarla en vez de un write final. Aplica directo a antu-ingest --research y a --rounds.
  • Descriptions con anti-gatillos — si algún día habilitas model-invocation en skills antu, el patrón "Use when… / Do NOT load to…" de OK es el estado del arte para que dos skills vecinas no compitan.
  • Versionado + estado de instalación — skill-state.yml permite diagnosticar "qué versión de la skill tiene este usuario"; tu antu-setup verifica presencia pero no versión.

Debilidades de OK que tú ya resolviste

  • Sin memoria de sesión — OK no tiene handoff: apuesta todo al CRDT + shadow repo. Para flujos de agente serial (tu caso de uso), tu tríada HANDOFF/HISTORY/PLAYBOOK con rotación y triage es estrictamente más capaz.
  • Sin análisis semántico de salud — el lint de OK es mecánico; no detecta contradicciones factuales entre docs ni propone merges con debate. Tu prune capa 2 (y kuyen-lint) no tienen equivalente allá.
  • Acoplamiento total al servidor — si el MCP no está, la suite entera degrada a una válvula de escape. Tu diseño fail-open con dependencias opt-in es más robusto para entornos heterogéneos.
  • Sin localización — monolingüe inglés; tu manejo conversación-vs-locale-del-vault no aparece en ningún lado de OK.
  • Sin sanitización de ingesta — OK ingiere fuentes verbatim sin chequeo de credenciales/inyección; tu antu-sanitize.sh cubre un riesgo que ellos no miran.

Riesgos de OK como referencia

La skill de plataforma de OK está al límite del presupuesto de contexto razonable (~2.600 palabras el núcleo, ~9.000 más en references) y su densidad — paréntesis anidados, incisos encadenados — apuesta a que el modelo lo procese todo. Es el extremo opuesto de la apuesta de Kuyen. Si tomas ideas de OK, toma los mecanismos (pressure-testing, anti-gatillos, degrees of freedom), no el volumen: su necesidad de 20.000 palabras nace de proteger un CRDT multi-escritor que tú no tienes.
